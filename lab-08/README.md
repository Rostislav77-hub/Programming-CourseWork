# Lab 08: Authentication Proxy for API Service

У цій лабораторній роботі реалізовано просунутий проксі-сервіс для обробки та аутентифікації HTTP-запитів. Архітектура побудована на класичних патернах проектування, що забезпечує слабку зв'язність (loose coupling) та високу масштабованість системи.

## Архітектурні патерни

1. **Decorator / Chain of Proxies (Декоратор)**
   Запити проходять через ланцюжок незалежних шарів (LoggingProxy → RateLimitProxy → AuthProxy). Кожен шар виконує своє завдання (Single Responsibility Principle) і передає запит далі. Це дозволяє легко додавати нові шари (наприклад, кешування) без зміни коду.
2. **Strategy (Стратегия)**
   Логіка впровадження облікових даних винесена в окремі класи (`ApiKeyStrategy`, `JwtStrategy`). Проксі не знає, як саме влаштована авторизація — він просто викликає метод apply() у поточної стратегії.
3. **Dependency Injection (Внедрение зависимостей)**
   Фінальний споживач ('ApiService') отримує готовий HTTP-клієнт через конструктор і нічого не знає про логування, ліміти або токені.

## Ключові фічі (Додаткові завдання)

* **Automatic Token Renewal:** Перехоплення помилки `401 Unauthorized` та автоматичне оновлення протухлого JWT з повторенням вихідного запиту.
* **Rate Limiting:** Захист від спаму (повертає `429 Too Many Requests` при перевищенні ліміту).
* **Logging:** Детальне консольне логування кожного перехопленого запиту та статусу відповіді.
* **Dynamic Switching:** Можливість зміни стратегії авторизації (API Key ↔ JWT) прямо під час роботи програми ("на льоту").
